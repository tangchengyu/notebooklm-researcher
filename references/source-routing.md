# 数据源能力矩阵与补采路由

## 路由原则

每轮以 `last30days` 的 `source_status` 为事实依据。这里的“覆盖”是能力上限，不代表本机本轮已经成功。

| 通道 | last30days 能力 | 默认补采规则 |
|---|---|---|
| Reddit | 原生 | `ok` 时不重复；失败/未配置时用 Web/其他社区补证 |
| X/Twitter | 可选 Cookie/API | 未获用户同意时不读取 Cookie；如缺失则保留覆盖缺口 |
| YouTube | 依赖 yt-dlp，可有字幕/评论 | `ok` 且包含字幕/摘要时不重复；只有元数据或失败时用 opencli 补字幕 |
| TikTok / Instagram | 通常需 ScrapeCreators | 只有用户重视且同意配置时启用；否则标缺口 |
| Hacker News | 原生、免密钥 | `ok` 时不重复 |
| Polymarket | 原生、免密钥 | 只对预测/市场类主题作为信号，不当作事实证明 |
| GitHub | 原生，优先 gh 认证 | `ok` 时不再跑仓库发现；需要 README/Issue/Release/全历史时定向深挖 |
| Web | 原生或宿主 Web 搜索 | 仍保留官网/标准/官方文档通道，因为广义 Web 不等于权威一手资料 |
| Digg / Techmeme | 可选 CLI | 成功时不重复相关新闻发现；关键事实仍回到原始页面 |
| arXiv | 可选 CLI、近窗口 | 只替代“近期 arXiv 发现”；不替代正式发表核验、奠基论文、其他数据库 |
| 小红书 | 可选 xhs 服务，需显式启用 | 只有状态 `ok` 且正文/评论充分时不补；否则用 opencli/社媒助手 |
| LinkedIn / Threads / Pinterest / Bluesky / StockTwits / Trustpilot | 条件启用 | 仅当主题相关且配置成功时采用；不为了数量强行开启 |
| B站 | 不原生覆盖 | 保留 opencli + SubBatch 通道 |
| 知乎 | 不原生覆盖 | 保留 opencli 通道 |
| Context7 | 不原生覆盖 | 软件/API/框架主题保留 |
| Bohrium / 专利 | 不原生覆盖 | 学术、工程、专利主题保留在 my-notebooklm-research |
| PubMed/PMC、Crossref、OpenAlex、CORE、Unpaywall | 不等价覆盖 | 按学科保留正式核验与合法全文通道 |

## 决策顺序

对每个平台依次判断：

1. 用户是否明确要求？明确要求优先，但仍需授权和合法访问。
2. `last30days` 是否支持？不支持则走补充通道。
3. 本轮 `source_status` 是否为 `ok`？不是则判断是否值得补采。
4. 成功结果是否包含正文/字幕/评论，而非只有元数据？内容不足时定向补全，不重复搜索发现阶段。
5. 是否与研究窗口一致？`last30days` 默认近 30 天；历史、奠基、长期比较必须补充。
6. 是否属于权威一手来源？官网、标准、法规、正式文档和论文应独立核验。

## 去重优先级

从强到弱：

1. DOI、arXiv ID、PMID、专利号、GitHub `owner/repo`、BV/视频 ID、平台帖子 ID；
2. 规范化 URL：主机小写、移除 fragment、去除 `utm_*`、`fbclid`、`gclid`、`spm` 等追踪参数；
3. 规范化标题：Unicode 归一化、空白压缩、去常见标点并小写；
4. 标题高度相似且作者/日期一致时人工合并。

不要因为摘要相似就删除不同来源。交叉验证需要保留独立来源。

## 合并策略

- 同一 URL 多次出现：合并来源标签、互动数据和互补摘要。
- 预印本与正式发表版：正式版作为主记录，保留 arXiv/开放版本链接与版本关系。
- 同一视频的元数据和字幕：合并为一条完整记录。
- 同一 GitHub 仓库的发现记录与 Release/Issue：仓库作为主实体，具体 Release/Issue 可作为独立子证据。
- 官网新闻稿与媒体转载：保留官网为一手来源；媒体如提供独立核查或背景可另留。

## 补采停止条件

满足以下条件后停止继续堆来源：

- 主要研究问题每个关键结论至少有一条高质量一手来源，或两条相互独立的可靠二手来源；
- 近期舆情、官方事实、技术实现和学术证据之间没有明显空白；
- 新增来源只重复既有观点，不能增加反例、限制、数据或解释；
- 达到 NotebookLM 项目容量或用户设定的时间/数量预算。
